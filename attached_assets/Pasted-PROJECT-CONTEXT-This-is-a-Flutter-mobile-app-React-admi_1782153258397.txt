FEATURE 1: SHARED PENDING POOL (Most Critical)
═══════════════════════════════════════════════════════════

LOGIC: 40 labours = ONE global pool per day across ALL sites.
Once marked at any site → vanishes from every other site's list.

FIRESTORE STRUCTURE (create/update):

Collection: labours/{labourId}
  - name: string
  - dailyWage: number
  - type: "regular" | "temporary"
  - isActive: boolean
  - phone: string (optional)
  - joiningDate: timestamp

Collection: attendance/{YYYY-MM-DD}/records/{labourId}
  - labourId: string
  - labourName: string (snapshot)
  - siteId: string
  - siteName: string (snapshot)
  - wageAtTime: number ← COPY from labour at time of marking, never reference live wage
  - remark: string (default "")
  - markedAt: timestamp
  - markedBy: string (uid)
  - isTemp: boolean
  - status: "present" | "absent"

Collection: sites/{siteId}
  - name: string
  - isActive: boolean
  - supervisorId: string

FEATURE 4: WAGE SNAPSHOT (Critical for Payroll Accuracy)
═══════════════════════════════════════════════════════════

RULE: attendance.wageAtTime is written ONCE at marking time.
Never update it. Never reference labour.dailyWage in reports.
All payroll calculations use wageAtTime only.

If admin changes a labour's wage today → old attendance records 
stay unchanged. Only future markings use new wage.
