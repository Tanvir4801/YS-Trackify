UPDATE FEATURE: QR Attendance to work with Site-wise flow

CONTEXT:
App already has QR scanning. Now labours are marked 
site-wise. QR must know WHICH site to mark attendance at.
Solution: Supervisor starts a "site session" first, 
then all scans during that session belong to that site.

═══════════════════════════════════════════════════
SESSION MODEL — NEW
═══════════════════════════════════════════════════

Create Firestore collection: attendanceSessions/{sessionId}
  - sessionId: auto
  - supervisorId: uid
  - supervisorName: string
  - siteId: string
  - siteName: string
  - date: string (YYYY-MM-DD local timezone)
  - startedAt: timestamp
  - endedAt: timestamp | null
  - status: "active" | "completed" | "abandoned"
  - markedCount: number (increments on each scan)
  - totalPresent: number
  - allowancesApplied: boolean (default false)

RULE: Only ONE active session per supervisor at a time.
RULE: Only ONE active session per site per day.
If supervisor tries to start session for a site that 
already has a completed session today → show warning:
"Site A attendance already completed today. 
 Open to add more or view details?"

FLUTTER — HOME SCREEN UPDATE
═══════════════════════════════════════════════════

Site cards on home screen:
Each card shows:
  - Site name (bold)
  - "X marked today" count
  - Status badge:
    PENDING (amber) — no session started
    IN PROGRESS (blue) — active session exists  
    COMPLETE (green) — session ended

    "Session active by [name] — view only"
  If COMPLETE → open summary / view only

FLUTTER — QR SCAN SCREEN
═══════════════════════════════════════════════════

Route: /attendance/scan/:sessionId

TOP BAR:
  🟢 [Site Name] — Session Active
  [X marked] [End Session button]

BODY — Two tabs:
  Tab 1: QR Scanner
  Tab 2: Manual List

QR SCANNER TAB:
  - Use mobile_scanner package (already in project 
    or add it)
  - Camera fills most of screen
  - Scan overlay: square bracket corners (not full box)
  - On successful scan:
    1. Vibrate once
    2. Parse QR JSON → extract labourId
    3. Validate: type == "labour_qr" AND 
       appId == current firebase project
    4. Check if already marked today (any site):
       YES → flash RED → show "Already marked at [site]"
       NO → flash GREEN → mark attendance
    5. Write attendance record with current sessionId+siteId
    6. Show bottom notification card (3 sec):
       "[Labour Name] ✓ marked at [Site]"
       Buttons: [Add Remark ✏] [Undo ↩]
    7. Camera immediately ready for next scan
    8. Increment session.markedCount

  DUPLICATE SCAN HANDLING:
  If same QR scanned twice in same session:
  Flash AMBER → "Already marked in this session"
  Do NOT write duplicate record
  
  INVALID QR HANDLING:
  If QR is not a labour QR (wrong format/app):
  Flash RED → "Invalid QR code"
  Brief vibration pattern (different from success)

MANUAL LIST TAB:
  Shows all labours NOT yet marked today
  Search bar at top
  Each row: [Avatar] [Name] [Mark Present button]
  This is fallback when QR is damaged/missing

═══════════════════════════════════════════════════
BUGS TO PREVENT
═══════════════════════════════════════════════════

1. QR scan must be debounced — 1 second cooldown 
   between scans to prevent double-scanning same code

2. Session must be tied to LOCAL date not UTC —
   same rule as attendance date field

3. If app crashes mid-session — session stays "active"
   On next open → detect orphaned active session →
   "Resume your session for Site A?" dialog

4. QR camera must release properly on screen dispose
   to prevent camera staying open in background

5. Offline scan support:
   If no internet → queue scans locally →
   sync when connection restored →
   show "X scans pending sync" indicator

6. Two supervisors cannot have active sessions 
   for the SAME site simultaneously →
   Second supervisor sees lock message
